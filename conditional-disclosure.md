# Conditional Disclosure — threshold viewing keys

**Status:** Draft v1 (testnet) · **License:** CC0-1.0 · **Requires:** [privacy-pool.md](./privacy-pool.md), [nullifier-registry.md](./nullifier-registry.md), [CSAP.md](./CSAP.md)

> Conditional disclosure lets a user prove facts about **one** of their private
> transactions to an authorized party — only when an **M-of-N custodian quorum**
> approves, and only when the transaction **qualifies** under an on-chain policy
> (amount above a threshold) — without ever exposing the full viewing key, which
> would reveal *all* of their transactions. The target users are enterprises:
> KYC/travel-rule providers, DAO treasuries with oversight boards, and audit firms.

## 0. Design decisions

Three decisions shape this spec; they are recorded here because each rejects a
plausible alternative.

**D-1: Shamir reconstruction is the recovery backstop, not the disclosure path.**
Plain Shamir reconstruction hands the complete viewing key to whoever combines
M shares, violating the acceptance requirement that *no single party ever holds
the full key*. The active disclosure path therefore uses **FROST threshold
signatures** — the group's secret key is generated by distributed key generation
and **never exists in one place, not even during signing**. Shamir shares of the
viewing key (§2) remain available as an explicitly-documented escrow backstop
for non-cooperative or key-loss scenarios.

**D-2: The custodian signature is verified on-chain, not in-circuit.**
Verifying a secp256k1 Schnorr signature inside a BN254 Groth16 circuit costs
hundreds of thousands of constraints (non-native field arithmetic) and would
blow the project's `pot16` ceiling; circomlib's cheap in-circuit EdDSA would
require FROST over Baby Jubjub, which no audited implementation supports. The
verifier contract/program checks a **BIP-340 Schnorr signature** natively
(Ethereum: the `ecrecover` trick; Solana: the `secp256k1_recover` syscall) over
a message bound to the proof's public `context` input. The circuit and the
signature meet at `context` — the same binding pattern `withdrawal.circom`
already uses.

**D-3: The v1 circuit statement is pool-scoped.**
"Amount above a threshold" is only a meaningful *hidden* predicate where amounts
are hidden — the privacy pool. Native stealth-payment amounts are already public
on-chain; what is private there is *linkage*, and proving DKSAP linkage
in-circuit requires secp256k1 EC math beyond `pot16`. v1 therefore proves
statements about **pool commitments** (all-Poseidon, a structural sibling of
`withdrawal.circom`); stealth-payment oversight is served by the §2 escrow.
In-circuit linkage disclosure is future work (§10).

## 1. Roles and keys

| Role | Holds | Notes |
|---|---|---|
| **Owner** | pool note openings `(value, label, nullifier, secret)`; CSAP viewing/spending keys | generates disclosure proofs |
| **Custodians** (N) | one FROST key share each | M-of-N co-sign disclosure authorizations; DKG per RFC 9591, ciphersuite FROST(secp256k1, SHA-256) with Taproot tweaking, so the group signature is standard **BIP-340** |
| **Requester** | nothing secret | the auditor/regulator/board receiving the disclosure; submits the on-chain transaction |

The FROST **group public key** `P` (32-byte x-only, even-Y per BIP-340) is
registered on-chain in a **disclosure policy** (§3). `P` is *not* the viewing
key and is unrelated to CSAP key material; it is an authorization key created
by the custodian DKG ceremony and its secret never exists whole.

## 2. Viewing-key escrow (Shamir backstop)

`splitViewingKey(viewingKey, m, n)` splits the 32-byte CSAP viewing-key scalar
(CSAP §2.1) into `n` shares with threshold `m` using Shamir secret sharing over
GF(256) (the independently audited `shamir-secret-sharing`). Each share is
wrapped in a versioned
envelope:

```
share = base64( 0x01 ‖ m ‖ n ‖ index ‖ raw_share )      // scheme "shamir-gf256-v1"
```

**Normative warning:** combining `m` shares **reconstructs the full viewing
key** and reveals the owner's *entire* incoming-payment history to the
combining party. This path is for key recovery and lawful non-cooperative
escrow only; it MUST NOT be used where the §4 proof path applies. After any
reconstruction event the owner SHOULD rotate keys (re-derive from a fresh
wallet signature and re-register via ERC-6538 / the stealth registry; the ONS
`sync` flow propagates rotation to names).

The spending key is never split. Shares grant read capability only.

## 3. Disclosure policy

A policy binds a custodian set to a pool and a qualification threshold:

```
Policy {
  pool        — the privacy-pool contract/program this policy can disclose from
  groupKeyX   — 32-byte x-only BIP-340 FROST group public key
  threshold   — minimum qualifying amount (wei / lamports), < 2^128
  m, n        — quorum descriptor (informational on-chain; M is enforced by FROST itself)
}
```

Policies are registered permissionlessly; `policyId` identifies one (Ethereum:
sequential id; Solana: policy PDA). A registered policy is immutable — rotation
means registering a new policy (a new DKG output) and retiring the old one
socially. Choosing *which* policy governs which funds is an application-layer
agreement between the owner and their oversight body; the chain enforces only
that a disclosure names a policy and carries that policy's quorum signature.

## 4. The disclosure statement — `conditional_disclosure.circom`

One Groth16 proof (BN254, Poseidon, depth-20 state tree — identical parameters
to `withdrawal.circom`) proving:

1. `commitment = Poseidon(value, label, Poseidon(nullifier, secret))` is a
   member of the pool **state tree** at `state_root` (depth-20 inclusion).
2. **Qualification:** `value > threshold` (both range-checked `< 2^128`).
3. **Disclosure nullifier:**
   `disclosure_nullifier = Poseidon(nullifier, context, DOMAIN_DISCLOSURE)`
   — one consumption per (commitment, context), see §7.
4. `value` and `label` are **public signals** — they *are* the disclosed data.
   `label` links the commitment to its `Deposit(commitment, label, value,
   leafIndex)` event, giving the requester the deposit provenance; `value` is
   the hidden amount being reported.
5. `context` is bound through constraint (3) (no dummy-square needed — the
   nullifier hash constrains it directly).

| Signal | Visibility | Notes |
|---|---|---|
| `nullifier`, `secret` | private | the commitment's openings |
| `state_siblings[20]`, `state_index[20]` | private | state-tree path |
| `value` | **public** | disclosed amount |
| `label` | **public** | disclosed deposit identifier |
| `threshold` | **public** | checked against the policy on-chain |
| `state_root` | **public** | must be a known pool root |
| `disclosure_nullifier` | **public** | consumed on-chain |
| `context` | **public** | binds policy/case/requester, §5 |

Public-signal order (snarkjs): `value, label, threshold, state_root,
disclosure_nullifier, context` — six signals, the same count as `withdrawal.circom`
(the Solana verifier shape carries over).

The proof does **not** include association-set membership: disclosure is
orthogonal to withdrawal compliance, and a requester investigating a tainted
deposit must be able to receive a disclosure about it.

A proof for a below-threshold note is **unsatisfiable** — the witness builder
fails; nothing about the note leaks. This is the "conditional" in conditional
disclosure: custodians can authorize a request *blind* (they sign `context`,
which contains no note data) and rely on the circuit to enforce qualification.

## 5. Authorization signature

**Message.** The signed message is the 32-byte big-endian encoding of the
`context` field element:

```
context = keccak256(encode(policyId, caseId, requester)) mod r        (r = BN254 scalar field)
m       = be32(context)
```

- Ethereum: `encode` = `abi.encode(uint256 policyId, bytes32 caseId, address requester)`.
- Solana: `encode` = `policy_pda(32) ‖ case_id(32) ‖ requester_pubkey(32)`.

`caseId` is an opaque requester-chosen identifier (legal case number, audit
engagement id). Custodians MUST display and review `(policyId, caseId,
requester)` before contributing a FROST signature share.

**Scheme.** FROST(secp256k1, SHA-256) with Taproot tweaking (Zcash Foundation
`frost-secp256k1-tr`), so the aggregate is a standard **BIP-340** signature
`(Rx, s)` over `m` verifiable against the x-only group key `Px`:

```
e = int( sha256_tagged("BIP0340/challenge", Rx ‖ Px ‖ m) ) mod n
valid ⟺ s·G == R + e·P        (R, P lifted with even Y)
```

**Ethereum verification (ecrecover trick).** With `n` the secp256k1 group
order, the contract computes `e` (SHA-256 precompile for the tagged hash) and
checks:

```
ecrecover( bytes32(n − mulmod(s, Px, n)),  27,  bytes32(Px),  bytes32(n − mulmod(e, Px, n)) )
   == address( keccak256(Rx ‖ Ry)[12:] )
```

which recovers the point `s·G − e·P` and compares it to `R`. The caller
supplies `Ry`; the contract requires `Ry` even and `(Rx, Ry)` on-curve, and
rejects a zero `ecrecover` result. `v = 27` always (BIP-340 keys are even-Y).

**Solana verification.** The same recovery with the `secp256k1_recover`
syscall (recovery id 0 = even Y), comparing the recovered 64-byte point to the
caller-supplied `(Rx, Ry)` directly; the tagged SHA-256 uses the `sol_sha256`
syscall.

## 6. Verifier interface

```solidity
event PolicyRegistered(uint256 indexed policyId, address pool, uint256 groupKeyX, uint128 threshold, uint8 m, uint8 n);
event Disclosure(uint256 indexed policyId, bytes32 indexed caseId, address requester,
                 uint256 label, uint256 value, bytes32 disclosureNullifier);

function registerPolicy(address pool, uint256 groupKeyX, uint128 threshold, uint8 m, uint8 n)
    external returns (uint256 policyId);

function disclose(
    Proof calldata proof,            // Groth16 (a, b, c)
    uint256[6] calldata signals,     // §4 order
    uint256 policyId,
    bytes32 caseId,
    SchnorrSig calldata sig          // (rx, ry, s)
) external;
```

`disclose` MUST, in order:

1. load the policy; require `signals[2] == policy.threshold`;
2. recompute `context` from `(policyId, caseId, msg.sender)` and require it
   equals `signals[5]` — the requester is the transaction sender, so a quorum
   authorization cannot be redirected;
3. verify the BIP-340 signature over `be32(context)` against `policy.groupKeyX` (§5);
4. require `IOpaquePrivacyPool(policy.pool).isKnownRoot(signals[3])`;
5. verify the Groth16 proof;
6. consume `signals[4]` in this verifier's nullifier registry (revert if present);
7. emit `Disclosure`.

The Solana program mirrors this: a `Policy` PDA, the pool account passed
read-only (owner-checked against the privacy-pool program, root checked against
its root history), an init-once `DisclosureNullifier` PDA, and an embedded
Groth16 verify (same `alt_bn254` syscall pattern as the pool's withdraw).

## 7. Nullifier domain

This closes the "Conditional disclosure" reservation in
[nullifier-registry.md](./nullifier-registry.md):

```
disclosure_nullifier = Poseidon(nullifier, context, DOMAIN_DISCLOSURE)

DOMAIN_DISCLOSURE = keccak256("opaque/disclosure/v1") mod r
                  = 2892858644728810973983554811705195156385130922452064297470708309156017996001
```

- The **arity-3 layout with a constant domain tag** is distinct from the pool's
  `Poseidon(nullifier)` and PSR's `Poseidon(stealth_pk, external_nullifier)`,
  per the registry's domain-separation rule.
- The registry lives in the **disclosure verifier** (per chain), never shared
  with the pool: disclosing a note MUST NOT spend it, and spending MUST NOT
  block disclosure.
- Consumption is per `(commitment, context)`: a second disclosure of the same
  note under a *different* case requires a fresh quorum signature and consumes
  a different nullifier — intended behaviour. Replaying the *same* disclosure
  is rejected.

## 8. Flow

```
setup:      custodians run FROST DKG (m-of-n) → group key Px
            registerPolicy(pool, Px, threshold, m, n) → policyId
            owner (optionally) splitViewingKey(m, n) → escrow shares to custodians

disclose:   requester opens case (policyId, caseId)
            owner builds witness for the target note; fails if value ≤ threshold
            custodians review (policyId, caseId, requester); M of them FROST-sign be32(context)
            owner hands proof + signals to requester
            requester submits disclose(proof, signals, policyId, caseId, sig)
            chain: policy ✓ context ✓ BIP-340 ✓ pool root ✓ Groth16 ✓ nullifier consumed
            → Disclosure event: (label, value) now provably on the record

recovery:   (exceptional) M custodians combine Shamir shares → full viewing key
            → full read access; owner rotates keys afterwards (§2)
```

## 9. Security considerations

- **No single point of key reconstruction (FROST path).** The DKG group secret
  never exists whole; signing shares are combined into a signature, not a key.
  The disclosed data is computed by the owner from note openings — the viewing
  key is not an input to the proof at all.
- **Blind authorization is safe.** Custodians sign only `context`; the circuit
  enforces qualification (`value > threshold`), so a quorum cannot be tricked
  into disclosing a non-qualifying note, and a malicious owner cannot disclose
  under a policy whose threshold the note does not meet.
- **Requester binding.** `context` commits to `msg.sender`/requester, so an
  intercepted proof + signature cannot be submitted by anyone else; replay of
  the same disclosure is blocked by the nullifier.
- **Signature malleability / wrong-key attacks.** BIP-340 verification rejects
  off-curve or odd-Y `R`, zero `ecrecover` output (Ethereum), and out-of-range
  `s ≥ n`. The tagged challenge hash domain-separates from every other Schnorr
  use of the key.
- **What the requester learns:** exactly `(value, label, deposit provenance via
  the Deposit event, qualification, case binding)` — nothing about other notes,
  the viewing key, or withdrawal destinations. Remainder commitments from
  partial withdrawals carry the same `label`, so a chain of partial spends is
  disclosable case-by-case but not globally linkable by the requester.
- **Owner cooperation is required** on the proof path (the owner holds the
  openings). Non-cooperation falls back to §2 escrow with its documented
  total-read-access cost — a deliberate, visible escalation rather than a
  silent capability.
- **Trusted setup.** Same `pot16` + Phase-2 posture as the pool; a production
  ceremony is a mainnet gate (audit plan).

## 10. Limitations and future work

- **Stealth-payment linkage proofs** (disclose "stealth address S is mine"
  without the viewing key) need secp256k1 scalar-mul in-circuit → a larger
  ptau and a dedicated audit; deferred.
- **Threshold decryption of the viewing key** (custodians decrypt *for one
  scanning session* without reconstruction) would soften the §2 backstop;
  deferred.
- **Value-band disclosure** (prove `value ∈ [a, b)` without the exact amount)
  is a trivial circuit variant; deferred until a requester needs it.
- **Custodian liveness** is social in v1: a dead quorum means no disclosures
  (and no recovery if shares are also lost). n should exceed m comfortably.
